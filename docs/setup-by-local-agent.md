# Setup playbook ŌĆö read this end-to-end before touching anything

> You are the bring-up agent for **beaver-designus** inside a corporate DS perimeter. Your job is to wire up two design systems (paths provided by the operator), adapt a corporate Qwen Code fork, and start the dev server. **You do this entirely through JSON configuration. You do not write any code. You do not edit any `.ts`, `.tsx`, `.js`, `.mjs`, or `.cjs` file.**
>
> If your reading of any step makes you want to edit a code file, **STOP. Re-read this document. The fix is in JSON.** If after re-reading the fix still requires code, write that conclusion into `docs/setup-report.md` and STOP. The maintainer will ship the fix upstream; the next agent re-runs from a fresh `git pull`.

---

## TL;DR ŌĆö the seven absolute rules

1. **You may edit exactly these five files / paths.** Everything else is read-only:
   - `manifest.config.json` ŌĆö DS configuration (paths, exclude lists, token grammar)
   - `runtimes.config.json` ŌĆö CLI runtime adapter overrides (corp Qwen / Claude fork). Copy from `runtimes.config.example.json` and edit. **This is where you adapt the corporate fork ŌĆö never `daemon/runtimes/defs/qwen.ts`.**
   - `.cache/` directory ŌĆö place DS clones / symlinks here
   - `manifest-overrides/<ds>/<package>.overrides.json` ŌĆö only files the scaffolder created with `TODO:` stubs, only when you have real content to fill in
   - `docs/setup-report.md` ŌĆö your output

2. **You never edit ANY of these:**
   - Anything under `daemon/`, `packages/`, `web/`, `shared/`, `tests/`, `skills/`
   - Anything ending in `.ts`, `.tsx`, `.js`, `.mjs`, `.cjs`
   - `tsconfig.json`, `tsconfig.dev.json`, `vite.config.ts`, `package.json`, `eslint.config.js`, `.prettierrc.json`
   - `manifest-data/*` (auto-generated by `npm run manifest:build`)
   - `packages/preview-runtime/src/component-map.ts` and `component-map.report.json` (auto-generated by `npm run preview:wire`)

3. **You never edit code.** Period. Not `daemon/runtimes/defs/qwen.ts`, not `daemon/runtimes/defs/claude.ts`, not `packages/manifest/src/scan/discovery.ts`, not anything else. To customize the Qwen adapter for a corporate fork, you write JSON into `runtimes.config.json` ŌĆö ┬¦1.6 below shows the exact shape and three worked examples.

4. **You never `git commit`, `git push`, or modify `.git`.** Leave the worktree clean for the operator to commit.

5. **You never `npm install <package>`** to "fix" a missing dependency. Missing transitives in DS repos are fixed by `npm run setup:ds`. Missing top-level deps are a maintainer issue ŌĆö write it in the report.

6. **You run seven npm commands in order.** That's the whole procedure. The only step where you read something and make a decision is ┬¦1.6 (Qwen adapter); even that is just choosing one of three pre-written JSON blocks.

7. **You produce exactly one output file: `docs/setup-report.md`.** If you wrote anything else (a script, a test, a patch), you went off-script. Delete it and re-read this document.

## ┬¦1 ŌĆö the seven commands

### 1.1 `npm install`

Installs project deps from `package.json`. Expected: ~30 s, no errors.

If it fails on a private `@platform-ui/*` or `@tui-react/*` package from the corporate registry:
- **Don't** add `--legacy-peer-deps`, `--force`, `--ignore-scripts`, `--registry <url>`, or any other flag.
- **Don't** create or edit `.npmrc`.
- Stop, write the exact error into the report, and ask the operator to provision the registry.

### 1.2 Place DS clones at `.cache/<id>`

For each DS:
```bash
mkdir -p .cache

# POSIX symlink to existing checkout:
ln -s <abs-path-to-ds> .cache/<ds-id>

# Windows junction:
New-Item -ItemType Junction -Path .cache\<ds-id> -Target <abs-path-to-ds>

# fresh git clone:
git clone <url> .cache/<ds-id>
```

The default `manifest.config.json` and `manifest.config.tbank.example.json` expect ids `beaver-ui` and `react-ui-kit`. If the operator's DSes have other ids, you'll update `manifest.config.json` `designSystems[*].source.localPath` in ┬¦1.3.

Verify:
```bash
test -f .cache/beaver-ui/package.json && test -f .cache/react-ui-kit/package.json
```

### 1.3 Choose / adapt `manifest.config.json`

**This is the only config file you ever edit.** Two paths:

**Path A ŌĆö T-Bank Beaver v0.244+ AND react-ui-kit Neodymium:**

```bash
cp manifest.config.tbank.example.json manifest.config.json
```

Done. Continue to ┬¦1.4.

**Path B ŌĆö anything else:** edit `manifest.config.json` directly. The fields, in order of how often they need to change:

| Field | What | Example |
|---|---|---|
| `designSystems[*].source.localPath` | Path to DS clone | `"./.cache/beaver-ui"` |
| `designSystems[*].componentRoot` | Where packages live INSIDE the DS | `"packages"` or `"packages/components"` |
| `designSystems[*].excludePackages` | Basename globs of non-UI packages **directly under `componentRoot`**. Dirs outside `componentRoot` (e.g. a sibling `tinkoff-packages/` legacy tree) are already not scanned ŌĆö listing them here is a harmless no-op, not a fix for anything. | `["analytics", "hooks", "core", "internal-*"]` |
| `designSystems[*].docsRoot` | Path(s) to MDX docs roots | `["auto-doc/docs/patterns"]` or `["docs", "packages/components"]` |
| `designSystems[*].tokenRoot` | Path to design-tokens package | `"packages/design-tokens"` (one DS only) |
| `designSystems[*].tokenAxisGrammar.pattern` | Regex matching the leaf-key vocabulary in `<token>.js` | `"^(?<surface>desktop\|ios\|android)(?<theme>Dark)?Value$"` |
| `designSystems[*].tokenCssVarPattern` | Template for emitted CSS var names | `"--{namespace}-{binding}-{variant}"` (default works for most DSes) |
| `designSystems[*].styles.globalStylesheets` | Relative paths (to DS root) of CSS the DS expects loaded ONCE ŌĆö reset/base/layout or an aggregated bundle. Without this, components render unstyled (a table collapses to one column). | `["dist/index.css"]` or `["styles/reset.css","styles/base.css"]` |
| `designSystems[*].styles.cssStrategy` | How the DS authors CSS. `auto` (default) detects it. Set explicitly if detection is wrong. | `"modules"` \| `"vanilla-extract"` \| `"linaria"` \| `"runtime-css-in-js"` \| `"auto"` |
| `designSystems[*].styles.postcssConfig` | Path (rel to DS root) to the DS's own PostCSS config, reused so nesting/mixins compile like the DS build | `"postcss.config.cjs"` |

`docs/audit-by-local-agent.md` ┬¦3 explains how to derive each of these from the DS source ŌĆö but if the operator has already run an audit and given you the values, use them directly.

Validate:
```bash
node -e "JSON.parse(require('fs').readFileSync('manifest.config.json'))"
```

### 1.4 `npm run setup:ds`

Runs `npm install --ignore-scripts --legacy-peer-deps --no-audit --no-fund` inside each `.cache/<ds>` so the DS's own transitive deps (react, classnames, date-fns, etc.) land in `.cache/<ds>/node_modules`. Vite resolves them from there at dev time.

Idempotent ŌĆö skips DSes that already have `node_modules` (use `npm run setup:ds -- --force` to re-install).

If a DS install fails:
- **Don't** edit the DS's `package.json` or `.npmrc`.
- **Don't** retry with different flags.
- Write the failure into the report and stop.

### 1.5 `npm run manifest:build`

Discovers packages, extracts prop signatures (including HOC patterns like `forwardComponent<E, Props>(cb)` and `createButton<Props>(hook)`), parses MDX + Storybook stories, extracts design tokens. Outputs to `manifest-data/`.

Expected log lines:
```
[manifest] [beaver] discovered N packages
[manifest] [beaver] indexed K MDX files across M root(s)
[manifest] [react-ui-kit] discovered N packages
[manifest] [react-ui-kit] extracted Z token groups
[manifest] wrote <total> entries to .../manifest-data
```

If `discovered 0 packages` for a DS that should have many: `componentRoot` in `manifest.config.json` is wrong. Fix it and re-run. **Don't edit any code in `packages/manifest/src/`** ŌĆö if you genuinely find a scanner bug (e.g. a re-export pattern isn't recognised), write a one-paragraph reproduction into `docs/setup-report.md` and stop.

### 1.6 Adapt the Qwen Code corporate fork via `runtimes.config.json`

**This is the only step where you THINK.** You read `<qwen-fork> --help` and edit one file: `runtimes.config.json`. **You do not edit any `.ts` file.** If you find yourself opening `daemon/runtimes/defs/qwen.ts`, close it ŌĆö the customization is JSON.

Start by copying the template:

```bash
cp runtimes.config.example.json runtimes.config.json
```

Now open `runtimes.config.json` and edit. The file is per-environment (gitignored) ŌĆö your changes live with the operator's machine.

#### Step 1 ŌĆö locate the binary

```bash
<operator-path-to-qwen> --version
```

If `<operator-path-to-qwen>` is NOT on PATH under its base name, the operator MUST set `QWEN_BIN=<absolute-path>` in the shell that runs `npm run dev`. Record this in the setup-report's "Env vars the operator must set" section.

If the fork's binary name differs from upstream `qwen` (e.g. `qwen-corp.exe` or `tbank-qwen`), put it in the `bin` field of the JSON.

#### Step 2 ŌĆö inspect the argv shape

```bash
<operator-path-to-qwen> --help | head -80
```

Look for:
- **Prompt input mode**: stdin (`-`)? Positional `--prompt <text>`?
- **MCP config flag**: `--mcp-config <path>` (upstream), `--mcp <path>`, `--tool-config <path>`, `--tools-config <path>`? The daemon writes a ready per-session `mcp.json` (with `BEAVER_DESIGNUS_SESSION_ID`/`PROJECT_ID` already baked in); your only job is to point the CLI at it via `${mcpConfigPath}` in `buildArgs`.
  - **If the fork has NO MCP argv flag at all** ŌĆö it ingests MCP servers ONLY from a settings file / fixed config location (e.g. `~/.<tool>/settings.json` `mcpServers`) and there is no way to pass a path on the command line ŌĆö then `runtimes.config.json` (which only customizes argv) **cannot** wire it. **STOP.** Record in the report under "Required next steps": *"Fork `<name>` exposes MCP only via `<the settings file/dir you found>`, no argv flag. The daemon currently delivers MCP as a per-session file path passed through argv (`${mcpConfigPath}`); a config-file delivery mode is a maintainer change, not a config one."* Do not hand-edit a global settings file (it has no per-session `SESSION_ID`/`PROJECT_ID` and will fail).
- **Auto-approve / yolo flag**: `--yolo` (upstream), `--auto-approve`, `--no-confirm`, `--non-interactive`? **If no such flag exists, STOP** ŌĆö the CLI will hang on every tool call. Write the gap into the report.
- **Stream output**: any JSONL/streaming flag? If the events emitted match Claude Code's shape (`{type: "assistant", message: {content: [...]}}` etc.), you can use `"streamFormat": "claude-stream-json"` (this is also what surfaces the model's *reasoning* as a separate streamed block in the chat; `"plain"` streams the answer text live but cannot separate reasoning). If unsure, leave as `"plain"`.
- **Coverage limit**: the JSON config covers any **argv** shape (flags, positionals, stdin) ŌĆö there is no upper bound on argv patterns. It does **not** cover forks that take MCP only via a non-argv settings file; that is the STOP case above, not something you can express here.

#### Step 3 ŌĆö write the `runtimes.config.json`

Three reference shapes follow. Place exactly ONE block under `runtimes.qwen` (or `runtimes.claude` if it's the Claude adapter you're customizing). Delete the others.

Available `${placeholders}` in `buildArgs`:
- `${mcpConfigPath}` ŌĆö absolute path to the per-session `.beaver-designus/mcp.json` file the daemon writes
- `${systemPromptFile}` ŌĆö absolute path to the per-session system-prompt file
- `${userMessageFile}` ŌĆö absolute path to the per-session user-message file
- `${allowedTools}` ŌĆö CSV like `mcp__beaver_designus__placeComponent,mcp__beaver_designus__setProp,...`
- `${allowedToolsSpaced}` ŌĆö space-separated variant for forks that take a variadic flag

Unknown placeholders pass through unchanged so misspellings are visible (you'll see the literal `${typo}` in the argv).

**Reference A ŌĆö fork is identical to upstream Qwen.** Don't ship `runtimes.config.json` at all; the built-in default applies.

**Reference B ŌĆö minimal override (renamed flags):**

```json
{
  "runtimes": {
    "qwen": {
      "displayName": "Qwen Code (corp)",
      "bin": "qwen-corp.exe",
      "buildArgs": [
        "--auto-approve",
        "--mcp", "${mcpConfigPath}",
        "-"
      ]
    }
  }
}
```

(`versionArgs`, `streamFormat`, `promptViaStdin`, `binEnvVar` all inherit from the built-in default since you didn't override them.)

**Reference C ŌĆö JSON-lines streaming + system-prompt-as-file:**

```json
{
  "runtimes": {
    "qwen": {
      "displayName": "Qwen Code (corp)",
      "bin": "qwen-corp.exe",
      "streamFormat": "claude-stream-json",
      "buildArgs": [
        "--auto-approve",
        "--mcp-config", "${mcpConfigPath}",
        "--system-prompt-file", "${systemPromptFile}",
        "--stream-format", "jsonl",
        "-"
      ]
    }
  }
}
```

Only use `"streamFormat": "claude-stream-json"` if the fork's JSONL events match Claude Code's keys (read `daemon/stream-format/claude-stream-json.ts` to verify ŌĆö you read it, you don't edit it). When in doubt, leave `"plain"`; tool-call events still arrive via the daemon's MCP server channel.

**Reference D ŌĆö fork takes the prompt as a positional file argument and doesn't read stdin:**

```json
{
  "runtimes": {
    "qwen": {
      "displayName": "Qwen Code (corp)",
      "bin": "qwen-corp",
      "promptViaStdin": false,
      "buildArgs": [
        "--auto-approve",
        "--mcp", "${mcpConfigPath}",
        "--prompt-file", "${userMessageFile}"
      ]
    }
  }
}
```

If your fork doesn't match any of AŌĆōD directly, mix-and-match the fields. The schema validates strictly ŌĆö unknown keys cause a clear error at daemon startup.

#### Step 4 ŌĆö auth env vars (no code, no checked-in secrets)

If the corporate fork needs `QWEN_API_KEY` / `CORP_LLM_TOKEN` / similar:
- **Don't** put them in `runtimes.config.json` (no env-var support there ŌĆö the schema is closed).
- **Don't** put them anywhere in the repo.
- Record in the setup-report's "Env vars the operator must set" section.
- The daemon forwards `process.env` (minus `CLAUDE_CODE_USE_POWERSHELL_TOOL`) to the spawned CLI, so anything in the shell that runs `npm run dev` reaches the fork.

#### Step 5 ŌĆö verify the override is loaded

```bash
node --import tsx -e "import('./daemon/runtimes/registry.ts').then(m => console.log(JSON.stringify(m.AGENT_DEFS.find(d => d.id === 'qwen'), null, 2)))"
```

Expect output reflecting your override (custom `bin`, function-style `buildArgs` derived from your JSON array). If you see the upstream-Qwen defaults instead:
- `runtimes.config.json` not in the repo root? It must be at the SAME level as `package.json`.
- JSON invalid? Run `node -e "JSON.parse(require('fs').readFileSync('runtimes.config.json'))"`. The schema also throws on unknown top-level keys.
- Override id wrong? Key under `runtimes` must be exactly `qwen` (or `claude`), lowercase.

### 1.7 `npm run preview:wire`

Does three things in order:

1. Writes `tsconfig.dev.json` extending `tsconfig.json` with DS-scope `paths` (e.g. `@beaver-ui/*` ŌåÆ `./.cache/beaver-ui/packages/*`). Auto-derived from `manifest.config.json` + each DS's first `package.json` `name`. **Don't hand-edit `tsconfig.dev.json`.**
2. Writes `packages/preview-runtime/src/component-map.ts` with **always-aliased** imports (`{ LabelDesktop as TuiReactCheckbox__LabelDesktop }`) so cross-package name collisions are structurally impossible. **Don't hand-edit `component-map.ts`.**
3. Runs `tsc --noEmit -p tsconfig.dev.json`. Distinguishes failure classes:
   - **TS2307 "Cannot find module"** ŌåÆ configuration issue. Generator surfaces a warning. **Fix is in `manifest.config.json`, not in component-map.ts.** Common cause: `setup:ds` didn't run, or `componentRoot` is wrong.
   - **TS2305 / TS2724 "has no exported member"** ŌåÆ real DS-side export mismatch. Generator drops the package and retries; the drop lands in `component-map.report.json`.

Expected:
```
[preview:wire] wrote tsconfig.dev.json + preview-aliases.json for N DS package(s)
[preview:wire] wrote E entries across P package(s); dropped D; warnings: 0
```

It may also print `M DS package(s) have NO loadable source entry` followed by a
package list. That means those packages ship built output only (their
`package.json` `main`/`exports` points into a `dist/` that does not exist
because `setup:ds` installs with `--ignore-scripts`) **or** they expose no
`src/index.*` / `"source"` field. This is a **DS-side packaging fact, not a
config error you can fix** ŌĆö `preview:wire` drops them from `component-map.ts`
on purpose so the dev server still boots. Copy the list (and each entry's
`tried:` paths) from `packages/preview-runtime/src/component-map.report.json`
into the setup-report's "Non-blocking deltas". Do **not** try to "fix" it by
editing code, building the DS, or rewriting `manifest.config.json`.

`dropped > 0` is otherwise fine ŌĆö same: copy each entry from
`component-map.report.json` into "Non-blocking deltas".

`warnings > 0` with TS2307 ŌåÆ re-do ┬¦1.3 (config) and ┬¦1.4 (`setup:ds`).

|---|---|---|
| `manifest-built` | `manifest-data/index.json` exists with entries | re-run ┬¦1.5 with corrected config |

| `ds-deps-installed` | every DS has `node_modules` | re-run ┬¦1.4 |

| `ds-styles-resolve` | declared `styles.globalStylesheets` exist; zero-runtime CSS strategy has its Vite plugin | declared-but-missing path ŌåÆ fix `styles.globalStylesheets` in `manifest.config.json`; "no global stylesheets" ŌåÆ add them (see ┬¦1.3 + audit doc); missing vanilla-extract/linaria plugin ŌåÆ **maintainer dep**, record in setup-report and continue |
| `typecheck-component-map` | `tsc -p tsconfig.dev.json` is clean | re-run ┬¦1.7 |

Each failure prints the concrete command (or instruction) to fix it. Run it,
re-run `npm run preview:doctor`, repeat until all eight say `ok` ŌĆö **except
`preview-entries-resolve`** and the **maintainer-dep half of
`ds-styles-resolve`** (a DS that needs the vanilla-extract/linaria Vite
plugin), which are informational: if the only remaining failures are dropped
DS packages with no loadable entry, or a missing zero-runtime CSS plugin,
that is an accepted degraded state ŌĆö record it in the setup-report and
proceed to ┬¦1.9. The "declared global stylesheet not found" failure IS
yours: it's a wrong path in `manifest.config.json` `styles.globalStylesheets`.

## ┬¦2 ŌĆö failure recipes (what to do, what NOT to do)

| Symptom | Cause | DO | DON'T |
|---|---|---|---|
| `npm install` fails on private package | corp registry not configured | ask operator for `.npmrc` to drop in | edit `.npmrc` yourself |
| `setup:ds` fails inside a DS | DS shrinkwrap / postinstall hook fires despite `--ignore-scripts` | write into report | retry with `--force --legacy-peer-deps --whatever`; edit DS package.json |
| `manifest:build` discovers 0 packages | `componentRoot` wrong | edit `manifest.config.json` `componentRoot` | edit `packages/manifest/src/scan/discovery.ts` |
| `manifest:build` errors in Stage 4b | token grammar mismatch | edit `manifest.config.json` `tokenAxisGrammar.pattern` | edit `packages/manifest/src/tokens/extract.ts` |
| `preview:wire` reports `TS2307` warnings | DS scope can't be resolved | re-run `setup:ds`, check `componentRoot` | edit `tsconfig.json` or `vite.config.ts` |
| Vite overlay: **`Failed to resolve import "@scope/pkg" from "ŌĆ”/component-map.ts". Does the file exist?`** | the wired DS package ships built output only (`main`/`exports` ŌåÆ an unbuilt `dist/`) or has no `src/index.*` / `"source"` field ŌĆö there is no loadable source to import | run `npm run preview:doctor`; copy its `preview-entries-resolve` output and the matching `dropped[].tried` entries from `component-map.report.json` into the setup-report "Non-blocking deltas"; if it blocks core components, set verdict ŌÜĀ’ĖÅ and STOP | edit `component-map.ts`, `vite.config.ts`, `tsconfig.dev.json`, the DS `package.json`, or run a DS build ŌĆö **this is a maintainer/DS-side fix, not yours** |
