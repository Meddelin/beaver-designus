# Setup playbook — read this end-to-end before touching anything

> You are the bring-up agent for **beaver-designus** inside a corporate DS perimeter. Your job is to wire up two design systems (paths provided by the operator), adapt a corporate Qwen Code fork, and start the dev server. **You do this entirely through JSON configuration. You do not write any code. You do not edit any `.ts`, `.tsx`, `.js`, `.mjs`, or `.cjs` file.**
>
> If your reading of any step makes you want to edit a code file, **STOP. Re-read this document. The fix is in JSON.** If after re-reading the fix still requires code, write that conclusion into `docs/setup-report.md` and STOP. The maintainer will ship the fix upstream; the next agent re-runs from a fresh `git pull`.

---

## TL;DR — the seven absolute rules

1. **You may edit exactly these five files / paths.** Everything else is read-only:
   - `manifest.config.json` — DS configuration (paths, exclude lists, token grammar)
   - `runtimes.config.json` — CLI runtime adapter overrides (corp Qwen / Claude fork). Copy from `runtimes.config.example.json` and edit. **This is where you adapt the corporate fork — never `daemon/runtimes/defs/qwen.ts`.**
   - `.cache/` directory — place DS clones / symlinks here
   - `manifest-overrides/<ds>/<package>.overrides.json` — only files the scaffolder created with `TODO:` stubs, only when you have real content to fill in
   - `docs/setup-report.md` — your output

2. **You never edit ANY of these:**
   - Anything under `daemon/`, `packages/`, `web/`, `shared/`, `tests/`, `skills/`
   - Anything ending in `.ts`, `.tsx`, `.js`, `.mjs`, `.cjs`
   - `tsconfig.json`, `tsconfig.dev.json`, `vite.config.ts`, `package.json`, `eslint.config.js`, `.prettierrc.json`
   - `manifest-data/*` (auto-generated by `npm run manifest:build`)
   - `packages/preview-runtime/src/component-map.ts` and `component-map.report.json` (auto-generated by `npm run preview:wire`)

3. **You never edit code.** Period. Not `daemon/runtimes/defs/qwen.ts`, not `daemon/runtimes/defs/claude.ts`, not `packages/manifest/src/scan/discovery.ts`, not anything else. To customize the Qwen adapter for a corporate fork, you write JSON into `runtimes.config.json` — §1.6 below shows the exact shape and three worked examples.

4. **You never `git commit`, `git push`, or modify `.git`.** Leave the worktree clean for the operator to commit.

5. **You never `npm install <package>`** to "fix" a missing dependency. Missing transitives in DS repos are fixed by `npm run setup:ds`. Missing top-level deps are a maintainer issue — write it in the report.

6. **You run seven npm commands in order.** That's the whole procedure. The only step where you read something and make a decision is §1.6 (Qwen adapter); even that is just choosing one of three pre-written JSON blocks.

7. **You produce exactly one output file: `docs/setup-report.md`.** If you wrote anything else (a script, a test, a patch), you went off-script. Delete it and re-read this document.

## §0 — what the operator gives you

1. **Two DS source locations**, each either an absolute path on disk or a git URL.
2. **Absolute path to the corporate Qwen Code fork binary** (e.g. `/opt/corp/bin/qwen-corp`).
3. **(Optional)** the names of any auth env vars the corporate Qwen fork needs.

## §1 — the seven commands

### 1.1 `npm install`

Installs project deps from `package.json`. Expected: ~30 s, no errors.

If it fails on a private `@platform-ui/*` or `@tui-react/*` package from the corporate registry:
- **Don't** add `--legacy-peer-deps`, `--force`, `--ignore-scripts`, `--registry <url>`, or any other flag.
- **Don't** create or edit `.npmrc`.
- Stop, write the exact error into the report, and ask the operator to provision the registry.

### 1.2 Place DS clones at `.cache/<id>`

For each DS:
```bash
mkdir -p .cache

# POSIX symlink to existing checkout:
ln -s <abs-path-to-ds> .cache/<ds-id>

# Windows junction:
New-Item -ItemType Junction -Path .cache\<ds-id> -Target <abs-path-to-ds>

# fresh git clone:
git clone <url> .cache/<ds-id>
```

The default `manifest.config.json` and `manifest.config.tbank.example.json` expect ids `beaver-ui` and `react-ui-kit`. If the operator's DSes have other ids, you'll update `manifest.config.json` `designSystems[*].source.localPath` in §1.3.

Verify:
```bash
test -f .cache/beaver-ui/package.json && test -f .cache/react-ui-kit/package.json
```

### 1.3 Choose / adapt `manifest.config.json`

**This is the only config file you ever edit.** Two paths:

**Path A — T-Bank Beaver v0.244+ AND react-ui-kit Neodymium:**

```bash
cp manifest.config.tbank.example.json manifest.config.json
```

Done. Continue to §1.4.

**Path B — anything else:** edit `manifest.config.json` directly. The fields, in order of how often they need to change:

| Field | What | Example |
|---|---|---|
| `designSystems[*].source.localPath` | Path to DS clone | `"./.cache/beaver-ui"` |
| `designSystems[*].componentRoot` | Where packages live INSIDE the DS | `"packages"` or `"packages/components"` |
| `designSystems[*].excludePackages` | Basename globs of non-UI packages | `["analytics", "hooks", "core", "internal-*"]` |
| `designSystems[*].docsRoot` | Path(s) to MDX docs roots | `["auto-doc/docs/patterns"]` or `["docs", "packages/components"]` |
| `designSystems[*].tokenRoot` | Path to design-tokens package | `"packages/design-tokens"` (one DS only) |
| `designSystems[*].tokenAxisGrammar.pattern` | Regex matching the leaf-key vocabulary in `<token>.js` | `"^(?<surface>desktop\|ios\|android)(?<theme>Dark)?Value$"` |
| `designSystems[*].tokenCssVarPattern` | Template for emitted CSS var names | `"--{namespace}-{binding}-{variant}"` (default works for most DSes) |

`docs/audit-by-local-agent.md` §3 explains how to derive each of these from the DS source — but if the operator has already run an audit and given you the values, use them directly.

Validate:
```bash
node -e "JSON.parse(require('fs').readFileSync('manifest.config.json'))"
```

### 1.4 `npm run setup:ds`

Runs `npm install --ignore-scripts --legacy-peer-deps --no-audit --no-fund` inside each `.cache/<ds>` so the DS's own transitive deps (react, classnames, date-fns, etc.) land in `.cache/<ds>/node_modules`. Vite resolves them from there at dev time.

Idempotent — skips DSes that already have `node_modules` (use `npm run setup:ds -- --force` to re-install).

If a DS install fails:
- **Don't** edit the DS's `package.json` or `.npmrc`.
- **Don't** retry with different flags.
- Write the failure into the report and stop.

### 1.5 `npm run manifest:build`

Discovers packages, extracts prop signatures (including HOC patterns like `forwardComponent<E, Props>(cb)` and `createButton<Props>(hook)`), parses MDX + Storybook stories, extracts design tokens. Outputs to `manifest-data/`.

Expected log lines:
```
[manifest] [beaver] discovered N packages
[manifest] [beaver] indexed K MDX files across M root(s)
[manifest] [react-ui-kit] discovered N packages
[manifest] [react-ui-kit] extracted Z token groups
[manifest] wrote <total> entries to .../manifest-data
```

If `discovered 0 packages` for a DS that should have many: `componentRoot` in `manifest.config.json` is wrong. Fix it and re-run. **Don't edit any code in `packages/manifest/src/`** — if you genuinely find a scanner bug (e.g. a re-export pattern isn't recognised), write a one-paragraph reproduction into `docs/setup-report.md` and stop.

### 1.6 Adapt the Qwen Code corporate fork via `runtimes.config.json`

**This is the only step where you THINK.** You read `<qwen-fork> --help` and edit one file: `runtimes.config.json`. **You do not edit any `.ts` file.** If you find yourself opening `daemon/runtimes/defs/qwen.ts`, close it — the customization is JSON.

Start by copying the template:

```bash
cp runtimes.config.example.json runtimes.config.json
```

Now open `runtimes.config.json` and edit. The file is per-environment (gitignored) — your changes live with the operator's machine.

#### Step 1 — locate the binary

```bash
<operator-path-to-qwen> --version
```

If `<operator-path-to-qwen>` is NOT on PATH under its base name, the operator MUST set `QWEN_BIN=<absolute-path>` in the shell that runs `npm run dev`. Record this in the setup-report's "Env vars the operator must set" section.

If the fork's binary name differs from upstream `qwen` (e.g. `qwen-corp.exe` or `tbank-qwen`), put it in the `bin` field of the JSON.

#### Step 2 — inspect the argv shape

```bash
<operator-path-to-qwen> --help | head -80
```

Look for:
- **Prompt input mode**: stdin (`-`)? Positional `--prompt <text>`?
- **MCP config flag**: `--mcp-config <path>` (upstream), `--mcp <path>`, `--tool-config <path>`, `--tools-config <path>`?
- **Auto-approve / yolo flag**: `--yolo` (upstream), `--auto-approve`, `--no-confirm`, `--non-interactive`? **If no such flag exists, STOP** — the CLI will hang on every tool call. Write the gap into the report.
- **Stream output**: any JSONL/streaming flag? If the events emitted match Claude Code's shape (`{type: "assistant", message: {content: [...]}}` etc.), you can use `"streamFormat": "claude-stream-json"`. If unsure, leave as `"plain"`.
- **Other forks** may have ENTIRELY different ways of passing MCP (e.g. a config file in a fixed location, or a `--tools-from <dir>` flag pointing at a folder of tool definitions). The JSON config covers any argv shape — there's no upper bound on the patterns supported.

#### Step 3 — write the `runtimes.config.json`

Three reference shapes follow. Place exactly ONE block under `runtimes.qwen` (or `runtimes.claude` if it's the Claude adapter you're customizing). Delete the others.

Available `${placeholders}` in `buildArgs`:
- `${mcpConfigPath}` — absolute path to the per-session `.beaver-designus/mcp.json` file the daemon writes
- `${systemPromptFile}` — absolute path to the per-session system-prompt file
- `${userMessageFile}` — absolute path to the per-session user-message file
- `${allowedTools}` — CSV like `mcp__beaver_designus__placeComponent,mcp__beaver_designus__setProp,...`
- `${allowedToolsSpaced}` — space-separated variant for forks that take a variadic flag

Unknown placeholders pass through unchanged so misspellings are visible (you'll see the literal `${typo}` in the argv).

**Reference A — fork is identical to upstream Qwen.** Don't ship `runtimes.config.json` at all; the built-in default applies.

**Reference B — minimal override (renamed flags):**

```json
{
  "runtimes": {
    "qwen": {
      "displayName": "Qwen Code (corp)",
      "bin": "qwen-corp.exe",
      "buildArgs": [
        "--auto-approve",
        "--mcp", "${mcpConfigPath}",
        "-"
      ]
    }
  }
}
```

(`versionArgs`, `streamFormat`, `promptViaStdin`, `binEnvVar` all inherit from the built-in default since you didn't override them.)

**Reference C — JSON-lines streaming + system-prompt-as-file:**

```json
{
  "runtimes": {
    "qwen": {
      "displayName": "Qwen Code (corp)",
      "bin": "qwen-corp.exe",
      "streamFormat": "claude-stream-json",
      "buildArgs": [
        "--auto-approve",
        "--mcp-config", "${mcpConfigPath}",
        "--system-prompt-file", "${systemPromptFile}",
        "--stream-format", "jsonl",
        "-"
      ]
    }
  }
}
```

Only use `"streamFormat": "claude-stream-json"` if the fork's JSONL events match Claude Code's keys (read `daemon/stream-format/claude-stream-json.ts` to verify — you read it, you don't edit it). When in doubt, leave `"plain"`; tool-call events still arrive via the daemon's MCP server channel.

**Reference D — fork takes the prompt as a positional file argument and doesn't read stdin:**

```json
{
  "runtimes": {
    "qwen": {
      "displayName": "Qwen Code (corp)",
      "bin": "qwen-corp",
      "promptViaStdin": false,
      "buildArgs": [
        "--auto-approve",
        "--mcp", "${mcpConfigPath}",
        "--prompt-file", "${userMessageFile}"
      ]
    }
  }
}
```

If your fork doesn't match any of A–D directly, mix-and-match the fields. The schema validates strictly — unknown keys cause a clear error at daemon startup.

#### Step 4 — auth env vars (no code, no checked-in secrets)

If the corporate fork needs `QWEN_API_KEY` / `CORP_LLM_TOKEN` / similar:
- **Don't** put them in `runtimes.config.json` (no env-var support there — the schema is closed).
- **Don't** put them anywhere in the repo.
- Record in the setup-report's "Env vars the operator must set" section.
- The daemon forwards `process.env` (minus `CLAUDE_CODE_USE_POWERSHELL_TOOL`) to the spawned CLI, so anything in the shell that runs `npm run dev` reaches the fork.

#### Step 5 — verify the override is loaded

```bash
node --import tsx -e "import('./daemon/runtimes/registry.ts').then(m => console.log(JSON.stringify(m.AGENT_DEFS.find(d => d.id === 'qwen'), null, 2)))"
```

Expect output reflecting your override (custom `bin`, function-style `buildArgs` derived from your JSON array). If you see the upstream-Qwen defaults instead:
- `runtimes.config.json` not in the repo root? It must be at the SAME level as `package.json`.
- JSON invalid? Run `node -e "JSON.parse(require('fs').readFileSync('runtimes.config.json'))"`. The schema also throws on unknown top-level keys.
- Override id wrong? Key under `runtimes` must be exactly `qwen` (or `claude`), lowercase.

### 1.7 `npm run preview:wire`

Does three things in order:

1. Writes `tsconfig.dev.json` extending `tsconfig.json` with DS-scope `paths` (e.g. `@beaver-ui/*` → `./.cache/beaver-ui/packages/*`). Auto-derived from `manifest.config.json` + each DS's first `package.json` `name`. **Don't hand-edit `tsconfig.dev.json`.**
2. Writes `packages/preview-runtime/src/component-map.ts` with **always-aliased** imports (`{ LabelDesktop as TuiReactCheckbox__LabelDesktop }`) so cross-package name collisions are structurally impossible. **Don't hand-edit `component-map.ts`.**
3. Runs `tsc --noEmit -p tsconfig.dev.json`. Distinguishes failure classes:
   - **TS2307 "Cannot find module"** → configuration issue. Generator surfaces a warning. **Fix is in `manifest.config.json`, not in component-map.ts.** Common cause: `setup:ds` didn't run, or `componentRoot` is wrong.
   - **TS2305 / TS2724 "has no exported member"** → real DS-side export mismatch. Generator drops the package and retries; the drop lands in `component-map.report.json`.

Expected:
```
[preview:wire] wrote tsconfig.dev.json with N DS scope alias(es)
[preview:wire] wrote E entries across P package(s); dropped D; warnings: 0
```

`dropped > 0` is fine — copy each entry from `packages/preview-runtime/src/component-map.report.json` into the setup-report's "Non-blocking deltas".

`warnings > 0` with TS2307 → re-do §1.3 (config) and §1.4 (`setup:ds`).

### 1.8 `npm run preview:doctor`

Preflight check. Runs six validations:

| Check | Verifies | Fix on failure |
|---|---|---|
| `manifest-built` | `manifest-data/index.json` exists with entries | re-run §1.5 with corrected config |
| `tokens-css` | `manifest-data/tokens.css` exists | re-run §1.5 |
| `component-map-current` | `component-map.ts` newer than `index.json` | re-run §1.7 |
| `ds-paths-exist` | every `source.localPath` resolves | re-do §1.2 |
| `ds-deps-installed` | every DS has `node_modules` | re-run §1.4 |
| `typecheck-component-map` | `tsc -p tsconfig.dev.json` is clean | re-run §1.7 |

Each failure prints the concrete command to fix it. Run the fix, re-run `npm run preview:doctor`, repeat until all six say `ok`.

### 1.9 `npm run dev`

Daemon on `127.0.0.1:7457`, vite on `127.0.0.1:5173`+.

```bash
curl -s -o /dev/null -w "vite=%{http_code} daemon=%{http_code}\n" http://127.0.0.1:5173/ http://127.0.0.1:7457/api/health
```

Both 200.

Smoke test:
```bash
SESSION=$(curl -s -X POST http://127.0.0.1:7457/api/sessions \
  -H 'content-type: application/json' -d '{}' \
  | python -c "import sys,json; print(json.load(sys.stdin)['sessionId'])")

curl -s -X POST http://127.0.0.1:7457/api/sessions/$SESSION/message \
  -H 'content-type: application/json' \
  -d '{"content":"Поставь PageShell как корень и закончи."}'

# in another shell:
curl -N http://127.0.0.1:7457/api/sessions/$SESSION/events
```

Expected SSE sequence: `prototype:set-root rev=0 root=null` → `status start` → one or more `status tool-call` → `prototype:set-root rev=1 root=<PageShell>` → `status end code=0`.

If no tool-call event in 30 s:
- Most likely: Qwen fork hangs waiting for tool confirm → §1.6 Step 3, your buildArgs is missing the auto-approve flag.
- Second-most likely: auth env vars not in the shell that started `npm run dev` → §1.6 Step 4.

## §2 — failure recipes (what to do, what NOT to do)

| Symptom | Cause | DO | DON'T |
|---|---|---|---|
| `npm install` fails on private package | corp registry not configured | ask operator for `.npmrc` to drop in | edit `.npmrc` yourself |
| `setup:ds` fails inside a DS | DS shrinkwrap / postinstall hook fires despite `--ignore-scripts` | write into report | retry with `--force --legacy-peer-deps --whatever`; edit DS package.json |
| `manifest:build` discovers 0 packages | `componentRoot` wrong | edit `manifest.config.json` `componentRoot` | edit `packages/manifest/src/scan/discovery.ts` |
| `manifest:build` errors in Stage 4b | token grammar mismatch | edit `manifest.config.json` `tokenAxisGrammar.pattern` | edit `packages/manifest/src/tokens/extract.ts` |
| `preview:wire` reports `TS2307` warnings | DS scope can't be resolved | re-run `setup:ds`, check `componentRoot` | edit `tsconfig.json` or `vite.config.ts` |
| `preview:wire` drops many packages with `TS2305`/`TS2724` | manifest names exports the DS doesn't actually have at package root | list them in setup-report "Non-blocking deltas" | edit `component-map.ts` to delete imports |
| Vite browser console "MIME type ''" | DS path not in fs.allow | confirm `source.localPath` exists; `vite.config.ts` auto-reads it | edit `vite.config.ts` |
| Smoke test times out without tool-call event | Qwen fork hangs on confirm OR auth env var missing | fix `runtimes.config.json` `buildArgs` (auto-approve flag) OR set env var in shell | edit `daemon/runtimes/defs/qwen.ts` or `load-overrides.ts` |
| Tree renders but every node is `UnknownComponentFallback` | component-map didn't generate; OR id mismatch | re-run `preview:wire`; check `component-map.report.json` | hand-edit `component-map.ts` |
| anything else | unknown | write into report, stop | improvise on code |

## §3 — final reminders

1. **Code = read-only.** `.ts`, `.tsx`, `.js`, `.mjs`, `.cjs` — read freely to understand, never modify.
2. **Config = JSON only.** `manifest.config.json` is your one editable config. Override file scaffolds in `manifest-overrides/<ds>/*.overrides.json` are also JSON, also yours to fill — only when scaffolded with `TODO:` markers.
3. **Single output.** `docs/setup-report.md`. Plus the auto-generated `component-map.report.json` from `preview:wire` (already written by the script — leave it).
4. **No git, no commits, no pushes.**
5. **Two reasonable retries per step**, then stop. If the same command fails twice with the same root cause, the fix is upstream — write it into the report.
6. **The maintainer ships upstream fixes.** Your job is to surface what's broken with enough detail that the next agent (running from a fresh `git pull`) finds the procedure works. Patches you write locally will be overwritten.
